# Troubleshooting

Lessons hard-won from building real decks. Skip this file unless something is misbehaving.

## Layout problems

### "A slide overflows in the browser preview"

The slide content exceeds the canvas dimensions (1123×794 for A4 landscape).

**Fix order**:
1. Count bullets. >6 at speaker density or >8 at reading density → split into two slides.
2. Image too big? Look at `max-height` in the CSS for that slide's layout. Default content slides cap images at 380 px height; the split layout caps the visual column at 600 px. If the image is portrait-orientation, drop the cap further (workbench photo in our reference talk used 460 px).
3. Heading too long? `h2` is 38 px by default — for very long titles, drop to 32 px on that slide only with an inline class.
4. Bullets too verbose? Check the user's outline — did you accidentally paraphrase or expand? Compare back to the source.

Don't fix overflow with `@media print` overrides. The screenshot-based PDF export means print CSS barely runs. Fix it in the screen CSS and the PDF inherits the fix.

### "The PDF doesn't match the browser"

If you're using the bundled `export_pdf.sh`, this should be impossible — the PDF is a screenshot. If you're seeing a mismatch:

- **Did the script complete normally?** If it errored part-way, the PDF may be missing slides.
- **Are fonts loading?** The script waits for `document.fonts.ready` but custom fonts via `@import` inside CSS sometimes don't trigger that signal. Move the font load to a `<link>` tag in `<head>`.
- **Are images on the slide?** Relative paths like `assets/photo.png` resolve correctly because the script serves the parent directory. Absolute filesystem paths (`/Users/name/photo.png`) won't work.

### "Bullets are overflowing only on certain slides"

Slides with longer bullets need either tighter spacing or a smaller font. Add a slide-specific class:

```html
<section class="slide tight">
```

…and target it in CSS:

```css
.slide.tight { padding: 40px 64px; }
.slide.tight h2 { font-size: 32px; margin-bottom: 12px; }
.slide.tight li { font-size: 18px; margin: 6px 0; }
.slide.tight .quote { font-size: 14px; margin: 4px 0 4px 16px; padding: 4px 12px; }
```

This is the pattern we settled on in the reference talk for slide A3 (four quote receipts).

## Font problems

### "Custom font not appearing in the PDF"

The screenshot waits 500ms after `document.fonts.ready` before capturing. If your font still doesn't show:

- Use Google Fonts with a `<link>` tag in `<head>` — most reliable
- If you're using a self-hosted font, it must be in the same directory tree as the HTML so the local server can serve it
- Check the browser console — the script prints any console errors

### "Fonts look different in the PDF"

The PDF font is whatever the screenshot captured. If the browser fell back to a system font (because the custom font failed to load), the PDF shows the system font.

## Image problems

### "Image is missing in the PDF"

The PDF export script starts a local HTTP server in the directory containing the HTML. Images must be accessible via relative paths from that directory.

- ✓ `<img src="assets/photo.png">` — works if `assets/photo.png` exists next to the HTML
- ✓ `<img src="../shared/logo.png">` — works if the path resolves under the served directory
- ✗ `<img src="/Users/me/Desktop/photo.png">` — absolute filesystem paths don't work
- ✗ `<img src="file:///...">` — same issue

If images appear in the browser but not the PDF, the user opened the HTML directly (`file://`) which has different path resolution than the export script's HTTP server. The fix is to use relative paths consistently.

### "SVG renders poorly"

SVGs are screenshot at the canvas pixel dimensions. If an SVG looks blurry, check that its `viewBox` is large enough — the `<svg>` element renders the viewBox content into whatever pixel size the layout gives it, and a small viewBox upscaled is fuzzy.

For inline diagrams generated by the skill, prefer absolute coordinates inside a `viewBox="0 0 1200 700"` or similar — large enough to look crisp at slide-size.

## Markdown parsing

### "The user's outline has a structure I haven't seen"

Common variants:

- **Numbered headings** (`### 1. The first slide`) — treat as `## Slide 1` equivalents
- **Hash-only structure** (`# Topic\n* bullet\n* bullet`) — flat, no slide breaks. Ask the user how to chunk it.
- **Indented bullets in a flat list** — the indented ones are sub-bullets of the previous bullet, render with nested `<ul>`
- **Lines starting with `>`** — pandoc-style speaker notes (`> *Speaker note:* ...`) or quote blocks. If they start with `*Speaker note:*` they go into `data-notes`. Otherwise they're quote blocks.
- **Wiki-link images** (`![[name.png]]`) — Obsidian syntax. Strip the brackets and look for the file in `assets/` or `images/`.
- **Multi-line bullets** — a bullet may span multiple lines if continuations are indented. Keep them as a single `<li>`.

### "The outline has 'A1', 'A2' style numbering — preserve it?"

Drop the numbering from the slide title (the deck has its own counter). But keep them in `<h3>` if the user clearly intends them as section markers (e.g., "A1 Introduction", "B1 The Problem" — A and B are sections).

## Mode B (enhance existing) gotchas

### "Existing deck uses different class names"

Check the existing HTML. If slides use `class="page"` or `class="reveal-slide"` instead of `class="slide"`, the skill's bundled assets won't compose with it. Either:

1. Adapt your edits to use the existing class names, OR
2. Tell the user the deck doesn't follow the standard convention and offer to migrate it (read the whole deck, restyle to `class="slide"`, and re-export)

### "Existing deck uses a different canvas size or no canvas"

If the existing deck doesn't use the fixed-canvas pattern at all (e.g., it's a Reveal.js deck or a one-shot SVG), don't try to retrofit it. Tell the user the structural change is bigger than just adding slides, and offer to rebuild from scratch.

## PDF export failures

### "First run errors out installing Playwright"

The script needs `npm` and a working network connection. Common fixes:

- Run manually: `npm install playwright && npx playwright install chromium`
- If behind a corporate proxy, set `HTTPS_PROXY` env var
- macOS: if `node` isn't found, `brew install node`

### "PDF is much larger than expected (~50 MB+)"

Each slide is a full-resolution PNG screenshot. For 30-slide decks at 1920×1080 this is normal. To shrink:

- Use `--compact` flag (renders at half resolution, ~60% smaller)
- Compress the PDF afterwards: `gs -sDEVICE=pdfwrite -dPDFSETTINGS=/screen -o small.pdf big.pdf` (requires Ghostscript)

### "Animations are missing in the PDF"

By design. The screenshot captures slides at their final visual state — animations don't survive. If the user needs animations, they need the HTML, not the PDF.
